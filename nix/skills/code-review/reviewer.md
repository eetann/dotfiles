# コードレビュー

## 実行手順

### 1. 差分の取得

派生元ブランチとHEADの差分を取得する。派生元ブランチはプロンプトで指定される（デフォルト: `development`）。

```sh
git diff <派生元ブランチ>...HEAD
```

### 2. 変更ファイルの読み込み

diffだけでは文脈が不足する場合、変更ファイルの該当箇所を読み込んで理解を深める。

### 3. レビュー

以下の8つの観点でレビューする。
全観点を網羅的にチェックする必要はない。コードに該当する問題がある観点だけ報告すること。

## レビュー観点

### 1. セキュリティ
- インジェクション（SQL、コマンド、XSS）
- 認証・認可の漏れ
- 秘密情報のハードコード・ログ出力
- 入力値の未検証

### 2. パフォーマンス
- N+1クエリ
- 不要な再レンダリング（React: useMemo/useCallback の不足・誤用）
- 巨大データの一括読み込み（ページネーションの欠如）
- 不要なループ・計算の重複

### 3. 保守性
- 6ヶ月後の自分が読めるか
- 関数・変数の命名が意図を表しているか
- マジックナンバー・マジックストリング
- 1関数が長すぎないか（目安: 50行以上は要注意）

### 4. エッジケース
- null / undefined の未処理
- 空配列・空文字列の考慮
- 並行処理・競合状態
- 境界値（0、負数、最大値）

### 5. 過剰設計
- YAGNI違反（今必要でない抽象化・設定項目）
- 不要なデザインパターンの適用
- 1回しか使わないヘルパー関数・ユーティリティ

### 6. エラーハンドリング
- 例外の握りつぶし（空のcatchブロック）
- 不適切なエラーメッセージ（デバッグ情報がない / ユーザーに内部情報が漏れる）
- リソースリーク（finally / close の欠如）

### 7. 型安全性
- `any` 型の乱用
- `as` による型断言の不適切な使用
- 型ガードの不足
- 型定義と実際の値の乖離

### 8. テスタビリティ
- テストしづらい密結合（外部依存の直接呼び出し）
- 副作用が関数の奥深くに埋もれている
- テストデータの組み立てが困難な構造

## 出力フォーマット

```markdown
## レビュー結果

### 🔴 Critical（すぐに直すべき）
- **[観点名]** `ファイルパス` — 問題の説明と修正案

### 🟡 Warning（直したほうがいい）
- **[観点名]** `ファイルパス` — 問題の説明と修正案

### 💭 Suggestion（余裕があれば）
- **[観点名]** `ファイルパス` — 提案内容

### ✅ Good（良い点）
- 良かったポイントを1〜3個挙げる
```

## ルール

- 問題がない観点は報告しない。「問題ありませんでした」は不要
- 修正案は具体的なコード例で示す
- 良い点も必ず挙げる（レビューは粗探しではない）
- 対象コードのプロジェクトの規約・パターンを尊重する
- 日本語で報告する
